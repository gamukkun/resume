### 取り組み
#### 自動テストシステムのCICDの改善
**状況**：
- 自身が入社した直後のミッションとして、CI/CDの改善を任されることになった。
- 配属チームにおいて、テストを実施せずにプルリクにマージしている状況に対し、課題感を感じていた。
- チームメンバーの中に知見者がおらず、導入工数観点でも、手を付けられずにいた状況であった。


**行動**：
- プルリクをマージする前にテストを実行し、テストを通過後にマージされるようチームへ提案し、自身で導入整備を行った。
- メンバーから開発効率が落ちるのでは、という懸念も上がったが、バグ修正や手戻りによるコストを伝えながら、納得感を持って取り組んでもらえるよう丁寧にコミュニケーションを取っていった。
- 作業手順をドキュメントにまとめ、自身がいなくとも対応できる環境を整え、属人化させないよう取り組んだ。


**結果**：
- 導入後、自動テストのエラーがほぼ発生しない状態に至った（プルリク10件に対し2-3回発生していたが、導入後は30件に0-1回程度）。
- メンバーからも、修正や再レビューの工数が削減ができたのは大きい、とフィードバックをもらえた。




#### JenkinsからGitHub Actionsへ自動テスト実行基盤の移行
**状況**：
- Jenkinsのセキュリティアラートに間に合わない等、自動テストの作業に集中できない状況が発生していた。
- 社内的な方針として、GitHub Actionsを導入していく方針であったものの、チーム内に知見を持つメンバーがおらず、導入に至っていなかった。


**行動**：
- JenkinsからGitHub Actionsの移行を推進。インフラを管理している別チームと連携し、必要なAWSのリソースを確保した
- 扱ったことのないIaCツールだったため、GitHubに上がっているリポジトリ、OSSにあがっているドキュメントを読みこみ、リポジトリ内のコードをみて書き方を学んだ上で、プルリクを作ってレビューを依頼。
- 横断的に関わるチームに所属しており、開発チームごとに決められたルールが異なるため、ルールの把握や、ルールに則って進めるのに苦労したが、ドキュメントを見に行ったり、メンバーにヒアリングして把握するようにした。また、ドキュメント上に足りないところがあれば加筆するように心がけた。


**結果**：
- 無事に移行が完了した。

### 取り組み
#### 自動テスト基盤のAWSアカウント移行
**状況**：
- 自動テスト実行基盤がプロダクト側のAWSアカウント上にあり、各所と連携を取る必要があり修正対応に時間がかかる状況があった。
 - Jenkinsのみ自チーム用のAWSアカウント上にあり、複数アカウント上で管理するのが手間であるとチーム内から意見がでていた。
- AWSの設定について一部コードで管理できておらず、実装者しか把握できていない状態であった。


**行動**：
- 社内におけるAWSの管理ルール、Terraformの実装ルール等の調査・確認を行った。
- 移行対象となる各種AWSリソース・設定を調査し、各メンバーが並行で進められるようタスクを作成し実施した。
- 移行する際に影響をうけるエンジニアへの周知・移行スケジュールの調整
- AWSやTerraformに関するスキルがないメンバーであってもタスクが進められるよう、ペアプロやチーム内で軽い勉強会を実施した。


**結果**：
- チームメンバーやAWSを全体的に管理しているSREチームから管理する手間が減ったとフィードバックをもらえた。
- チーム内で停滞していたインフラ系のタスクが取り組みやすくなった。
- 管理できていなかったAmazon EC2などのリソースの整理ができ、コスト削減にも繋げられた。
- AWSの設計やTerraformなどの実装レビューをメンバーも実施できるようになった。




#### PRベースでE2Eテストを実行する環境といった技術検証(EKS)
**状況**：
- 通常はリリース直前 (本番デプロイ時) にE2Eテストを実施しバグが出ていないかを確認していたが、テスト実行から確認、修正等の作業に時間がかかっていた。
- 開発段階でE2Eテストをより楽に実行できる仕組みがあれば、リリース時の作業時間を削減できるとチーム名で考えた。
- プロダクトはEKSで動いたため、ある程度知見があると想定し研究開発を行うことになった。


**行動**：
- SREチームと合同で開発を進めていたため、自身はインフラコスト計算やテストの実行時間等の計測などを主に実施していた。
- GitHub Actions + シェルスクリプトを用いるシステムとなっていたため、各種スクリプトの修正対応を行った。
- 研究開発としての位置づけであったため、メンバーとともにパイロットユーザー (お試ししてくれる方)を探し、依頼やレクチャーなどを実施した。


**失敗と学び**：
- システムとしてはある程度使えるものにはなっていたが、想定していたユースケースに対するユーザーのニーズの低さや、効果と運用コストが釣り合わず、本運用されることなくシステム開発が終了することになった。
- 開発続行・終了の判断を行う方 (オーナーシップを持ってやる方)を設定しておらず、開発ばかりに時間をかけてしまった。


システムとしては終了することになったが、課題設定のありかたやシステム開発自体の難しさ、1つの方法に固執せず様々な観点で開発を行っていくことを学ぶことができた。




#### 複数の検証環境のDB相乗り
**状況**：
- 自動テスト以外の課題解決も行っていこうと当時のチームリーダーとDBRE (Database Reliability Engineering) チームの合同で進めることになった。
- 実施理由としては、1つのAWSアカウントで作成できる上限に近づいており、コストの増大を抑える目的があった。
- プロダクト毎にDBのセットアップ手順が異なっているためそれらの整理を行う必要もあった。
- 各検証環境は各プロダクトチームが使っており、都度調整して進める必要があった。


**行動**：
- 設定もれなどが発生したため、Jiraチケットなどで細かくタスク化を行った。
- AWSなどの権限的に一部の社員しか作業ができず想定よりも作業期間が長くなってしまうことと、属人性が高くなってしまうことを避けるために細かい権限制御ができるよう提案などを行った。
- DB作成後の確認手段が統一されていなかったため、ドキュメントに書き起こすなどの対応を行った。


**結果**：
- チームメンバー全員でもれなくDB相乗り作業を進めることができた。 (想定の2倍早く完了)
- 各プロダクトチームでバラバラになっていたDBのセットアップ手順をまとめることができた。




#### 自動テスト基盤およびスクリプトの運用保守・改善
**詳細**：
- 数百以上のシナリオが手動または定期的に実行される中で、テストの成功率が70%以下のテストも多々あった。 (不安定なテストがある)
- EC2を用いて定期的に実行するE2Eテスト環境のログを毎朝確認していたが、ログや実行時のスクリーンショットを確認するのが手間で時間がかかっていた。 (Slack上に失敗したと通知が来るだけ)


**行動**：
- 週次で不安定なテストについての原因などをチームメンバーに共有し、適宜スクリプトを実装した人にも情報を共有し、修正対応依頼 or ペア作業の提案 or  自分たちで修正を行う等の対応を実施した。
- 自分で不安定なテストの改善する際
  - 過去実行ログや実行時のスクリーンショットを確認して、不安定になっている要因の仮説を立てる
  - 実際にテストを実行し、要素等の変更があればプルリクエストなどを確認して必要であれば実装した人に確認を行いつつ、修正を行うようにしていた。


- E2Eテスト環境ログについては、有識者にアドバイスをもらいながらログやスクリーンショットをSlackのスレッドに送れるよう設計し、実装した。


**結果**：
- テスト環境の実行ログ確認については、1〜2時間ほどかかっていたがメンバー平均20〜40分にまで
減らすことができた。

### 各プロジェクトにおける取り組み
#### コンテナ移行・インフラのコード化・CICDなど導入のリード
**状況**：
- 検証環境と本番環境でインフラ設定の差異があり、動作などが異なる場合があった。
- 開発者の手元でも環境を作成したいという要望が上がっていた。
- 実装したコードの反映作業も時間がかかり、自動化などをしたいとチームから意見が出ていた。


**行動**：
- コンテナ移行については、有識者がいたためその方と協力し、アプリに必要なミドルウェアなどを確認してDockerファイルに等に落とし込んだり、手元でDockerの操作ができるようドキュメントの整備などを行った。
- インフラのコード化に関しては、TerraformやAWS Cfnなどいくつかツールがあったので技術選定から行った。その中で、ロールバックなどが発生しないことと、今後他のクラウドサービスでの利用も考えて、terraformを利用することを決めた。
- 既存の設定をterraformに落とし込みつつ、不要な設定などを整理したうえでコード化を進めた。
- CICDについては、初めはAWS CodeBuildなどを参考ドキュメントなどを用い調査しながら設定をすすめた。その後、GitHubActionsがリリースされた際は全社的にGitHubActionsを使う方針に切り替え導入を進めた。


**結果**：
- 本番環境との差分をなくしつつ、各開発者の手元で環境を作成できるようになり、開発効率をあげることができた。
- CICD環境を整備したことで、リリースや検証環境への反映時の作業ミスを減らしつつ、作業時間を減らすことができた。
- 過去のインフラ設定などを見直すことで、適切なスペックやサーバー台数の調整などがしやすくなった。
- 作業を行ったことで、新たに必要な課題を見つけることができた
  - 監視の設定やアプリケーションのボトルネックを確認する手段の確立など



#### 新規開発に向けた共通Terraformモジュールの作成リード・インフラ操作CLIの作成
**状況**：
- 同時期にインフラを構築する要件があった。
- ECSなどを使っていたのだが、直接サーバーに入りログや設定値などを確認したい要望が上がっていた。
- 当時phpmyadminなどを使ってDBのデータを確認していたが、レコード数がなどが多く別のツールを使いたい要望が上がっていたが、外部から直接操作することができなかったため踏み台サーバーなどを手軽に用意できる環境を整備する必要があった。


**行動**：
- 必要となるインフラ構成の洗い出しを行った。
- 社内用の共通基盤となるものを目指したため、運用方法やドキュメント作成も適宜行った。
- 開発者からの要望などを聞いて、必要となる操作や困っていることなどをまとめた。
- ツールを作成するうえで必要な技術選定を有識者と共に行った。(最終的にGoを使ってCLIツールを作成することに決めた)


**結果**：
- 共通モジュールを使って素早くインフラを構築できるようになった。
- 開発者自身が共通モジュールの改善なども提案してくれ、常に改善ができるようになった。


**失敗と学び**：
- CLIツールについては、初めは便利に使ってもらえたが一部のメンバーにしか利用されなかった。
- 開発期間が特に決められていなかったため、改善などに時間をかけすぎてしまった。
- 必要最低限の機能を作った上で、メンバーに使ってもらい改善点を出すという繰り返し作業を行えば、より使われるツールとなったのではないかと思っており、今後は作りすぎずまずは試してみる・周知の方法を考えていく必要があると思った。


#### レンタルサーバーからAWSへの移行に関しての調整、設計・実装
**状況**：
- エイプリルフールなどのイベント時はゲームのHPに対して瞬間的にアクセスが多くなり、レンタルサーバーでは落ちてしまうことがあった。
- ゲーム外での施策として盛り上げる目的もあり、HPが見れなくなってしまうことは全体的な機会損失につながるとされていた。
- レンタルサーバーに必要なコードや画像などをアップロードしていたのだが、Gitなどで管理されていない・誰がいつ更新したかわからない状態であった。


**行動**：
- Gitで管理できるようソースコードをすべて移行した。
- Gitの操作が初めてというメンバーばかりであったため、操作に関するドキュメントや軽めの勉強会などを実施した。
- WordPressで実装されていたため、それに合わせる形でAWS上にEC2やRDSなどを使ってインフラの構築を行った。
- 負荷試験等の環境も用意し、想定ユーザーのアクセスに耐えきれるインフラかどうか確認を行った。
- CodeDeployなどを用いたCICDの環境を整備し、変更作業の自動化と誰が実施したのかわかるようにした。


**結果**：
- 適切にコードを管理できるようになった。
- 更新作業などにかかる時間を短縮することができた。
- 数万リクエスト/分が来た際もサーバーは落ちることなく稼働できるようになった。

## やりたいこと・目指したいこと

AWSやGCP等のクラウドサービスを用いたインフラの構築や運用・保守、社内開発チームや企業等への技術支援を行いたいと考えております。
